# Lumio Clinical Summary — Change Log

All changes made during the v3.0 prompt engineering and validator development cycle.
Root cause of every change: doctor-annotated review of a Singh (27 May 2026) Follow-Up
Psychiatric Consultation note generated by the v2.0 system.

---

## 1. Prompt Architecture — v2 → v3

### Change
Replaced the single monolithic shared prompt (`lumio_clinical_summary_system_prompt.txt` +
`lumio_dynamic_sections_3_4_prompt_v2.txt`) with a two-file composition per session type:
- `prompts/core_rules_v3.txt` — universal guardrails for all 7 types
- `prompts/type_prompts/<Type>.txt` — type-specific sections, fields, and vocabulary

### Why
The v2 prompt was ~29,000 characters and identical for all 5 psychiatric types. The model
had access to Follow-Up vocabulary when generating Intake notes and vice versa. Field rules
for one session type were contaminating output for another. A Room Change note was receiving
the full psychiatric vocabulary it would never need. Splitting into per-type modules meant
each type receives only the rules and vocabulary it actually uses.

---

## 2. New Rule: OMIT — DO NOT ANNOTATE (empty field rule upgrade)

### Change
Added the PATTERN RULE — ABSENCE BY ANY WORDING to `core_rules_v3.txt`.
The v2 rule had a finite list of forbidden phrases ("Not assessed", "Not discussed", etc.).
The v3 rule is structural: if a field's content *announces* the field has no real content —
in any wording, any tense — the entire field must be omitted.

Added explicit forbidden examples from real output failures:
- "no further detail on baseline sleep quality discussed"
- "not reported this session" (the "this session" hedge)
- "not addressed in detail"

### Why
The model kept generating new phrasings of the same meta-comment that weren't in v2's
forbidden list. A structural test ("does this state a clinical finding, or announce that
documentation is incomplete?") catches all variants, including ones not yet seen.

---

## 3. New Rule: FORMULATION CONTAINMENT

### Change
Added a dedicated `<formulation_containment>` block to `core_rules_v3.txt` defining
formulation (clinician's pattern recognition, trait attribution, ego-structure reasoning,
"consistent with X" language) and specifying it must live ONLY in the formulation field
(Clinician's Impression / Assessment).

Listed forbidden opening patterns verbatim:
- "Clinician noted [trait]..."
- "Egosyntonic [trait]..."
- "Patient tends to..."
- "Consistent with [diagnosis] ego-structure..."

### Why
The model was importing the clinician's personality/trait interpretation into Interpersonal
Relationships, MSE Insight, and target symptom fields. The doctor flagged entries like
"Clinician noted egosyntonic traits..." appearing in Interpersonal Relationships. Without
a named rule, the model had no basis to know this content was misplaced.

---

## 4. New Rule: VISUAL CUES (expanded)

### Change
Expanded the audio-only rule in `core_rules_v3.txt` to explicitly list every forbidden MSE
element: Appearance, Behaviour, Eye contact, Psychomotor activity, AFFECT (quality, range,
intensity, reactivity, congruence). Named specific forbidden phrases: "grief-like affect",
"anxious affect", "blunted affect", "tearful as observation".

Added the same prohibition with identical wording to all 7 type-specific modules.

Intentionally omitted affect vocabulary from Follow-Up's `<controlled_vocabularies>` — the
model cannot pick from a list it was never given.

### Why
The doctor marked affect observations as forbidden (audio-only session). The v2 rule said
"audio only" but did not name the specific MSE elements. The model kept generating affect
descriptions because it had no explicit list of what was forbidden.

---

## 5. New Rule: QUOTATION DISCIPLINE (hard cap)

### Change
Added a hard cap to `core_rules_v3.txt`: at most ONE verbatim quote per section (not
one-to-two — exactly one). Defined exactly 4 allowed categories:
1. Suicidal/self-harm ideation, plan, intent, or means
2. Abuse disclosure (historical or current)
3. Patient-stated rating (e.g. "6 out of 10")
4. Clinician-stated risk level or risk denial

Added a forbidden-phrases list with explicit paraphrase substitutions:
- "I feel scared everywhere" → "reports pervasive fear"
- "my mental strength was very poor" → paraphrase or omit (past content)
- Any description of how travel or medication felt → routine narrative, paraphrase

### Why
The doctor annotated "Excess verbatims again" pointing to a travel narrative quote:
*"It felt that way because this time when I travelled, the fear was not like that day —
it was a little less, but it was there."* This is routine emotional narrative, not in
any of the 4 allowed categories. v2 had general guidance to paraphrase but no hard cap
and no named forbidden examples.

---

## 6. New Rule: DIRECT EVIDENCE ONLY

### Change
Added `<direct_evidence_only>` block to `core_rules_v3.txt`: a field may be populated
only by transcript content that is ABOUT that field. Named the canonical violation:
taking a night dose during a panic episode is adherence context — it must NOT fill a
Sleep field.

### Why
The model was using incidental mentions from one clinical context as evidence for a
different field. A panic episode that happened to occur at night was being used as
"sleep evidence." The rule makes cross-field inference explicitly forbidden.

---

## 7. New Rule: TEMPORAL PRECISION (formalised)

### Change
Added `<temporal_precision>` to `core_rules_v3.txt` defining three bins explicitly:
- CURRENT (since last contact) → current-status and symptom fields
- PAST events → history/stressors/past-episode fields
- FUTURE plans → Plan section only

### Why
The model was putting future plans (upcoming travel, scheduled procedures) into current
functioning fields (Academic/Occupational). The temporal bins give the model a named
framework for deciding which field owns which content.

---

## 8. New Rule: PRE-OUTPUT CHECKLIST

### Change
Added an 8-point `<pre_output_check>` gate to `core_rules_v3.txt` that the model walks
before returning the note. Covers: empty fields, formulation containment, quotations,
visual cues, section renumbering, medication naming, one-fact-one-place, temporal precision.
Described as a "hard gate — non-compliance is not acceptable."

### Why
After multiple prompt-tightening iterations, violations kept reappearing in new phrasings.
A self-review gate forces the model to re-read its own output and catch violations before
returning, rather than relying solely on the generation-time rules.

---

## 9. Controlled Vocabularies — Scoped to Type

### Change
In v2, all controlled vocabularies (§3.1–§3.71) lived in one shared dynamic file loaded
for all types. In v3:
- Each type module embeds only its relevant vocabulary subset
- Follow-Up gets §FU.status/adherence/response/sleep/appetite/energy/med_change/risk_level
  + full MSE vocabulary
- Intake gets §onset/course/episode/consequences/substance/family_psych/functional
- Emergency gets §onset/risk.type/risk.ideation/risk.action only
- Regular Therapy and Couples Therapy have no formal vocabulary blocks (inline only)
- Room Change and Section Change have zero vocabulary

### Why
In v2 every type had access to all vocabulary. Follow-Up could accidentally use
`§onset: Insidious` (an Intake word) or `§risk.action: Hospitalisation arranged`
(an Emergency word). Scoping vocabulary to the type prevents cross-type contamination.

---

## 10. §4 Screening Checklists — Dropped

### Change
The v2 dynamic file's §4 diagnosis-driven coverage checklists (Depression, Anxiety, OCD,
Psychosis, Trauma, Substance Use, Universal) were not carried into v3.

### Why
The checklists instructed the model to ensure specific domains were covered for a given
diagnosis. This conflicted with v3's core principle of "document only what was discussed
in this session." The checklists pushed the model to fill fields even when the clinician
didn't cover that domain, producing empty-field disclaimers or fabricated content. The
template structure itself defines the available fields; the empty-field rule handles the
rest.

---

## 11. Temperature Set to 0

### Change
`ChatAnthropic(temperature=0)` in `app.py`.

### Why
To make generation deterministic. The same transcript should always produce the same note.
Temperature above 0 introduces randomness that makes it harder to identify whether a
prompt change actually fixed a violation.

---

## 12. Follow-Up Type Prompt — Side Effects Placement

### Change
Added explicit instruction in §2 Medication Review: a patient's worry that a vaccine or
injection caused symptoms is NOT a side effect — it belongs in Stressors/Triggers or is
noted as a misattribution the clinician addressed. Do not place it in Side Effects.

### Why
The doctor annotated the Side Effects field: "x S/E R/A not relevant here." The vaccine
episode (patient queried whether rabies vaccine caused weakness) was appearing in Side
Effects. The general rule ("side effects of medication only") was not specific enough;
naming the vaccine/injection pattern explicitly removed the ambiguity.

---

## 13. Follow-Up Type Prompt — Risk Level Clinician-Assigned Only

### Change
Added to §9 Plan: Risk Level label (Low/Moderate/High) may appear ONLY if the clinician
spoke that exact word in the transcript. If not, lead with the clinician's actual statement.
Never infer the level from the absence of ideation.

### Why
The model was generating "Low" risk level based on the absence of suicidal ideation rather
than the clinician assigning that label. The doctor flagged this. Same class of problem as
the response-label rule: a clinical assessment label must come from the clinician, not be
model-inferred.

---

## 14. Follow-Up Type Prompt — MSE Speech Guidance

### Change
Added to §7 MSE: include Speech only if the clinician noted a specific audible feature
(rate, tone, volume, quantity). "Engaged at length", "talked freely", "asked repeated
questions" are NOT Speech findings — the last is Thought Process. If no specific audible
feature noted, omit Speech entirely.

### Why
The model was filling the Speech field with filler content ("engaged at length") and
duplicating Thought Process content ("asked repeated questions on the same themes") under
Speech. Both are violations — one is an empty-field meta-comment dressed as content, the
other is a placement error.

---

## 15. Follow-Up Type Prompt — Insight Forbidden Patterns

### Change
Added to §7 MSE Insight: listed forbidden formulation-reconstruction patterns verbatim:
- "Clinician noted patient has difficulty trusting clinical guidance..."
- "Consistent with OCD ego-structure / psychology / structure..."
- "Patient demonstrated some awareness of her patterns..."
- Any range like "Grade 4–5"
Instruction: if you find yourself writing these, OMIT the Insight field. Use an exact
§MSE.insight value or omit.

### Why
The model was reconstructing an insight grade from formulation language rather than the
clinician's explicit assessment. "Grade 4–5 range" is an invented range (forbidden). The
forbidden patterns are the specific phrasings the model kept using; naming them explicitly
blocks them.

---

## 16. LLM Validator Agent (`validator.py`)

### Change
Created `validator.py`: a post-generation second-pass gate using Sonnet 4.6 at temperature 0.
Takes the generated note, reviews it against 13 rules, and returns a revised summary plus a
list of corrections made. Integrated into `app.py` — generator produces a draft, validator
revises it, UI shows the revised note with an expandable corrections list.

Validator uses LLM-only (no regex layer).

### Why
After multiple prompt-tightening iterations, the same violations kept reappearing in new
phrasings at temperature 0. This is the prompt-layer ceiling: more wording cannot reliably
catch phrasings that haven't been named yet. An LLM validator makes clinical judgments
("is this content tangential?", "is this formulation?") that pattern-matching cannot make.
Uses Sonnet 4.6 (same tier as the generator) because the validator is the final gate — a
weaker model cannot reliably catch what a stronger model slipped through.

---

## 17. Validator Rule R12 — Response Label Clinician-Assigned Only

### Change
Added R12 to `validator.py`: Response category labels (Good response / Partial response /
Minimal response / Worsening symptoms / Unable to assess) may appear ONLY if the clinician
spoke that exact term. If model-inferred, remove the label and lead with paraphrase only.

Also added the same rule to `Follow_Up_Psychiatric_Consultation.txt` §2 Medication Review.

### Why
The doctor circled "Partial response" in the output. The model assigned this classification
label from the patient's description of benefit — the clinician never used those words.
Same class of problem as Risk Level (rule R10): a clinical assessment category must be
clinician-stated, not model-inferred.

---

## 18. Validator Rule R9 Strengthened — Specific Duplication Patterns

### Change
Strengthened R9 (ONE FACT, ONE PLACE) in `validator.py` with named duplication patterns
the model repeatedly produced:
- Panic episode described in full in Subjective Report AND Response to Medications
- Reassurance-seeking described in Anxiety Symptoms AND MSE Thought Process
- Vaccination episode appearing in full in both Side Effects and Stressors

For each: named the primary home field and the action (remove or reduce to brief reference).

### Why
The general "each fact in one field" rule was not catching these specific cross-section
duplications. The validator was seeing them pass because R9 was too abstract. Naming the
exact patterns gives the validator concrete anchors to detect and fix them.

---

## 19. Validator Rule R13 — Temporal Precision

### Change
Added R13 to `validator.py`: current-status and functioning fields must contain only
current content. Future plans must appear ONLY in Plan or Stressors. If a current-status
field contains a future plan, move the plan content and remove it from the current-status
field.

### Why
"Upcoming travel home to Jharkhand planned around convocation period" was appearing in
Academic/Occupational Functioning — a current-status field. This is future content. The
prompt-layer temporal precision rule wasn't catching it in generation; adding it to the
validator ensures it gets caught and moved in post-processing.
